# Source Fuzzing Tools Testing

White-box, coverage-guided fuzzers driven by a per-target **harness** that lives
in the workspace source tree, one engine per language already covered by SAST.
They sit between SAST and DAST in the methodology: once SAST flags a suspect
function, you write a tiny harness that feeds fuzzer-generated bytes into that
function and let the engine search for a crashing input, confirming the bug in
the code before any live testing.

Fuzzing campaigns almost always exceed the MCP timeout, so `run_fuzzing_scan`
returns `status: backgrounded`; poll `get_fuzzing_report` to collect the crash.
Each crash artifact becomes one finding; `results[].input_base64` is the exact
triggering input (base64-encoded) and `stack_trace` / `log_tail` carry the
engine output.

## Canary validation (run first, per engine)

> Like the Chromium `xss-game` canary, these are deterministic crashes. They
> confirm the engine runs, the harness wires up, and a crash round-trips into the
> report. If a canary does **not** report `crash_found: true`, fix the engine
> before trusting it on real targets.

Each canary crashes the instant the first input byte is `'A'` (`0x41`), a
condition coverage-guided fuzzers solve in milliseconds. Drop the harness into a
workspace, then run `run_fuzzing_scan` with the listed config and confirm the
report's `results[0].input_base64` decodes to bytes starting with `A`.

**Atheris** (`harness.py`) - `config: {"harness": "harness.py", "max_total_time": 30}`

```python
import sys
import atheris

def TestOneInput(data):
    if len(data) >= 1 and data[0] == 0x41:  # 'A'
        raise RuntimeError("canary: first byte was 'A'")

atheris.Setup(sys.argv, TestOneInput)
atheris.Fuzz()
```

**Jazzer** (`FuzzCanary.java`, compile with `javac FuzzCanary.java` first) - `config: {"target_class": "FuzzCanary", "classpath": ".", "max_total_time": 30}`

```java
public class FuzzCanary {
    public static void fuzzerTestOneInput(byte[] data) {
        if (data.length >= 1 && data[0] == 'A') {
            throw new RuntimeException("canary: first byte was 'A'");
        }
    }
}
```

**Jazzer.js** (`fuzz.js`) - `config: {"harness": "fuzz.js", "max_total_time": 30}`

```js
module.exports.fuzz = function (data /* Buffer */) {
  if (data.length >= 1 && data[0] === 0x41) {
    throw new Error("canary: first byte was 'A'");
  }
};
```

**PHP-Fuzzer** (`target.php`) - `config: {"harness": "target.php", "max_total_time": 30}`

```php
<?php
/** @var PhpFuzzer\Fuzzer $fuzzer */
$fuzzer->setTarget(function (string $input) {
    if (strlen($input) >= 1 && $input[0] === "A") {
        throw new \Error("canary: first byte was 'A'");
    }
});
```

> Note: php-fuzzer 0.0.11 only detects `\Error` (and subclasses) as crashes.
> `\RuntimeException` and other `\Exception` subclasses are silently swallowed.
> Always use `throw new \Error(...)` in php-fuzzer harnesses.

"suspect function" column is the kind of input-handling sink worth harnessing.

## Suggested workflow

1. Create a workspace for the repo (`create_workspace` with the GitHub URL). For
   Jazzer.js install the target's `node_modules`, and for Jazzer compile the fuzz
   target into the workspace so its `classpath` resolves.
2. Run `detect_languages`, then `fuzzing_list_tools` with the same `workspace_id`
   and confirm only the language-appropriate engine appears (e.g. Atheris for the
   Python repo, Jazzer for WebGoat; Atheris must **not** appear for a PHP repo).
3. Add a harness next to the suspect function (see canaries for the per-engine
   shape) that decodes the fuzzer bytes into the function's real argument types.
4. `run_fuzzing_scan` with `config.harness` (or `target_class`/`classpath` for
   Jazzer), a `max_total_time` budget, and an optional seed `corpus`.
5. Poll `get_fuzzing_report`. On `crash_found: true`, base64-decode
   `results[].input_base64` to reproduce the triggering input locally.

## Chaining workflow: SAST to Source Fuzzing to DAST

1. **SAST** (Semgrep / CodeQL / Bandit) - flag a tainted-input sink (injection,
   deserialization, path traversal) and note the function and its parameter types.
2. **Source Fuzzing** - write a harness that feeds fuzzer bytes into exactly that
   function, then `run_fuzzing_scan`. A crash is in-code proof the sink is
   reachable with attacker-controlled input, plus a concrete triggering payload.
3. **DAST** (once wired) - replay the crash-derived input against the live
   endpoint that calls the function to confirm the vulnerability end-to-end.

*Converts a static "this looks dangerous" finding into a confirmed crashing input
before a single packet is sent at a live target.*
